## Архитектура

### Общая схема

Сервис состоит из двух процессов: API и Consumer.

API принимает HTTP запросы, атомарно записывает платёж и запись из outbox в PostgreSQL, и возвращает `202 Accepted`.  
Код 202 выбран нарочно, как сигнал что "процесс запущен, и вы можете делать что-то дальше без ожидания подтверждения".  
В примерах есть пример запроса поллинга.

Также есть фоновый поллер, который периодически читает непубликованные записи из таблицы `outbox` и отправляет их в RabbitMQ очередь `payments.new`.  

Consumer выставлен отдельным процессом, подписанным на очередь `payments.new`. Получив сообщение, он эмулирует обработку платежа, обновляет статус в БД, и отправляет webhook уведомление на URL указанный клиентом. При неудаче webhook доставки — повторяет с экспоненциальной задержкой. После исчерпания всех попыток публикует сообщение в `payments.dlq`.

---

### Outbox Pattern

Наивная реализация выглядит так: создаём платёж в БД, потом публикуем сообщение в RabbitMQ. Проблема — между этими двумя операциями процесс может упасть. Платёж создан, сообщение не отправлено, обработки не будет.

Outbox pattern решает это через атомарность: платёж и запись в таблицу `outbox` создаются в одной транзакции. Либо оба записались, либо ни один. Отдельный поллер читает непубликованные записи из `outbox` и отправляет их в RabbitMQ, и отмечает как опубликованные.

```sql
-- атомарно:
INSERT INTO payments (status='pending', ...)
INSERT INTO outbox (published_at=NULL, ...)
COMMIT

-- поллер каждые N секунд:
SELECT * FROM outbox WHERE published_at IS NULL FOR UPDATE SKIP LOCKED

-- После чего публикуем to RMQ, и... :

UPDATE outbox SET published_at = now()
```

Директива `FOR UPDATE SKIP LOCKED` — защита от гонки если в будущем появится несколько инстансов API. Каждая строка захватывается только одним поллером.

Поллер у нас является асинхронной задачей (asyncio.Task). Но это выбрано нарочно, для тестового задания.  
В продакшне его стоит вынести в отдельный процесс или заменить на CDC (Debezium). Соответствующее "TODO" имеется.

### Idempotency

Каждый запрос на создание платежа обязан содержать ключ `Idempotency-Key`. Он хранится в БД с уникальным индексом.

При повторном запросе с тем же ключом сервис возвращает существующий платёж без создания нового. Это защищает от дублей при сетевых ретраях на стороне клиента — клиент может безопасно повторять запрос сколько угодно раз.

### Retry и DLQ

Webhook доставляется через `httpx` с экспоненциальной задержкой между попытками:

попытка 1 → ошибка → ждём 1s
попытка 2 → ошибка → ждём 2s
попытка 3 → ошибка → ждём 4s
...
попытка N → ошибка → publish to payments.dlq

Важное разделение ответственности: статус платежа (`succeeded` / `failed`) отражает результат процессинга в шлюзе, а не факт доставки webhook. Платёж может быть `succeeded` при недоступном webhook URL. Это семантически корректно — платёж прошёл, уведомление не доставлено.

Сообщения в `payments.dlq` содержат `payment_id` и `reason` для ручного разбора.

---

### Два процесса, одна кодовая база

uvicorn app.main:app     → API + outbox poller
python -m app.workers    → consumer

Оба процесса шарят одни и те же модели, схему БД, настройки и инфраструктурный слой. Разделены только точки входа и ответственность: API принимает запросы и гарантирует публикацию, consumer обрабатывает и уведомляет.

### Схема БД
###### payments
id               UUID, PK
idempotency_key  VARCHAR, UNIQUE
amount           NUMERIC(18, 2)
currency         ENUM(RUB, USD, EUR)
description      VARCHAR
metadata         JSONB
status           ENUM(pending, succeeded, failed)
webhook_url      VARCHAR
created_at       TIMESTAMPTZ
processed_at     TIMESTAMPTZ

##### outbox
id               UUID, PK
payment_id       UUID, FK -> payments.id
created_at       TIMESTAMPTZ
published_at     TIMESTAMPTZ  <- NULL пока не опубликовано
